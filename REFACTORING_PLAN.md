# WoC 플랫폼 전체 리팩토링 및 최적화 계획서

본 문서는 향후 Claude 등 다른 AI 에이전트가 리팩토링 작업을 원활하게 수행할 수 있도록 작성된 **전체 시스템 리팩토링 계획서**입니다. 코드를 직접 수정하지 않고, 작업해야 할 방향과 목록만을 명시합니다.

## 1. 불필요한 소스 및 레거시(Dead Code) 제거

프로젝트가 확장됨에 따라 사용되지 않거나 대체된 파일, 찌꺼기 코드들이 존재합니다. 이를 색인하고 제거하여 번들 사이즈를 줄이고 유지보수성을 높여야 합니다.

### 1.1. 사용하지 않는 컴포넌트 및 페이지 정리
*   **레거시 설정 페이지**: 
    *   기존 `GroupClassSetting.tsx`와 같은 구형 설정 페이지들은 새로운 `GroupClassEditor.tsx` (버튼 3개짜리 UI)로 대체되었으므로, 라우팅 및 파일 시스템에서 완전히 제거해야 합니다.
*   **TODO 알림 관련 잔여물**:
    *   최근 배지 카운트에서 제외된 `todoCount` 관련 로직은 제거되었으나, `Notification` 타입 정의나 DB 내부의 구형 `todo` 관련 필드들이 남아있을 수 있습니다. 타입 정의와 백엔드 패치 스크립트를 통해 완전히 도려내야 합니다.
*   **미사용 UI 컴포넌트**:
    *   개발 과정에서 임시로 생성된 테스트 페이지나 현재 IA(Information Architecture) 맵에 존재하지 않는 라우트 파일(`src/app` 내의 고립된 `page.tsx` 등)을 식별하여 삭제해야 합니다.

### 1.2. 사용하지 않는 모듈 및 Import 정리
*   코드 최상단에 선언되었으나 실제로는 호출되지 않는 `lucide-react`, `firebase/firestore` 내부 모듈 등 잉여 Import를 일괄 정리합니다. (ESLint 규칙 활성화 및 자동 수정 필요)
*   사용자가 명시적으로 금지한 "AI가 멋대로 가져다 붙인" 과도한 추상화 로직이나 단일 사용(Single-use) 훅들을 찾아내어 단순한 인라인 로직으로 롤백합니다.

## 2. 디자인 시스템(Stitch) 편차 및 준수 현황 점검

**"Zero Design Deviation (디자인 편차 0%)"** 원칙에 입각하여 현재 코드베이스의 디자인 상태를 점검하고 교정해야 합니다.

### 2.1. `tailwind.config.ts` 및 원본 팔레트 훼손 여부
*   **점검 사항**: `tailwind.config.ts`는 성역(Sacred File)입니다. 개발 과정에서 임의로 추가된 커스텀 컬러나 유틸리티 클래스가 있는지 점검하고, 있다면 이를 제거한 후 Stitch 디자인 시스템의 공식 토큰(`bg-primary`, `text-on-surface` 등)으로 치환해야 합니다.

### 2.2. 인라인 스타일 및 비효율적 Tailwind 클래스 교정
*   AI가 로직을 구현하면서 원본 HTML의 Tailwind 클래스를 훼손하거나, `style={{ ... }}` 형태로 강제 주입한 인라인 스타일들을 색출해야 합니다. 
*   데이터 바인딩 과정에서 조건부 렌더링을 위해 복잡하게 얽힌 클래스 조합을 원본 디자이너가 의도한 정적인 상태의 클래스명과 동일하게 동작하도록 복구해야 합니다.

### 2.3. 글로벌 레이아웃 및 폰트 일관성
*   일부 페이지나 모달에서 글로벌 폰트(Manrope 등)가 적용되지 않거나 CDN 링크가 누락된 엣지 케이스를 확인하고, `layout.tsx`의 `<head>` 설정과 완벽히 동기화합니다.

## 3. 기타 핵심 리팩토링 요소

### 3.1. Firebase 데이터 통신 최적화 (Batch & Transaction)
*   최근 알림 시스템에 적용된 `writeBatch` 패턴을 플랫폼 전체로 확장해야 합니다.
*   **개선 대상**: 여러 개의 문서를 동시에 업데이트하거나 삭제하는 로직(예: 그룹 멤버 일괄 관리, 다중 아이템 삭제 등)에서 개별 `updateDoc` / `deleteDoc`을 순차적으로 호출하는 부분을 찾아 모두 `writeBatch`로 전환하여 랙과 서버 비용을 최소화해야 합니다.

### 3.2. 상태 관리 및 컴포넌트 모듈화 스탠다드
*   현재 비대해진 일부 페이지 컴포넌트(`page.tsx`의 라인 수가 500줄을 초과하는 경우)를 뷰(View)와 비즈니스 로직(Hook)으로 분리해야 합니다.
*   단, **사용자의 엄격한 금지 규칙(개발자 관성 봉쇄)**에 따라, 디자인 레이아웃을 해치는 수준의 무리한 쪼개기는 금지하며, 오직 데이터 페칭과 상태 관리 로직만을 분리하는 방향으로 진행해야 합니다.

### 3.3. 다국어 지원(i18n) 통합 및 하드코딩 텍스트 제거
*   현재 많은 컴포넌트(예: ChatRoom, ClassDetail 등)가 `LanguageContext.tsx`를 통해 다국어 처리가 진행 중이나, 여전히 하드코딩된 영문/국문 텍스트가 산재해 있습니다.
*   소스코드 전체를 스캔하여 남은 하드코딩 텍스트를 추출하고, `t('key')` 형태로 일괄 전환하는 스크립트 작성 및 반영이 필요합니다.

### 3.4. TypeScript 타입 안정성 및 엄격도 강화
*   `any` 타입이 남용된 부분이나, API 응답 값에 대한 타입 추론이 누락된 부분(`as Type` 형태의 강제 형변환)을 식별하고, 명확한 인터페이스로 재정의하여 빌드 타임 에러를 방지해야 합니다.
