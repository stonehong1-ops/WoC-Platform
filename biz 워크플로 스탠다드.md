# 비즈니스 워크플로우 스탠다드 (Biz Workflow Standard)

이 문서는 World of Community (WoC) 플랫폼 내의 커머스(Shop) 및 C2C 거래와 관련된 **인터랙션부터 구매 완료까지의 상태 변화 및 UI/UX 처리 워크플로우 표준**을 정의합니다. 향후 유사한 비즈니스 프로세스(예: Rental, Stay, Class 등)를 구축할 때 이 문서를 기준으로 삼아 일관성을 유지합니다.

---

## 1. 핵심 상태 정의 (Core States)

사용자가 상품(또는 서비스)에 관심을 보이고 실제 구매에 이르기까지의 과정은 플랫폼 내 위시리스트(Wishlist/Likes) 데이터를 기반으로 관리되며, 크게 세 가지 상태로 분류됩니다.

1. **`liked` (관심/찜)**
   - **트리거:** 사용자가 상품 상세 페이지에서 'Like(하트)' 버튼을 클릭했을 때.
   - **의미:** 상품에 단순히 관심을 표현한 상태.
   - **UI 표현:** 위시리스트 트레이에서 `LIKED ITEMS`로 분류되며, 기본 브랜드/카테고리 태그로 표시됩니다.

2. **`pending` (구매 협의 중)**
   - **트리거:** 사용자가 상품 상세 페이지에서 'Chat with Seller' (판매자와 대화하기) 버튼을 클릭했을 때.
   - **의미:** 구매 의사를 가지고 판매자와 소통을 시작한 상태.
   - **UI 표현:** 
     - 해당 대화창이 풀 팝업으로 노출됩니다.
     - 위시리스트 FAB(Floating Action Button)에서 `PENDING` 카운트가 증가하며, 리스트 최상단(또는 `in_progress` 다음)에 정렬됩니다.
     - 뱃지 디자인: `bg-primary` (메인 테마 컬러), 텍스트 `PENDING`.

3. **`in_progress` (주문 완료 및 진행 중)**
   - **트리거:** 사용자가 구매 절차(Order Summary)를 거쳐 결제 및 주문을 완료(Submit)했을 때.
   - **의미:** 실질적인 거래가 성사되어 상품 준비, 배송 또는 서비스가 진행 중인 상태.
   - **UI 표현:**
     - 구매 완료 화면에서는 복잡한 메뉴를 제거하고 오직 'Go to Shopping' (쇼핑 계속하기) 단일 액션 버튼만 제공하여 UX를 간소화합니다.
     - 위시리스트 FAB에서 `IN PROGRESS` 카운트가 증가하며, 리스트의 **가장 최상단(1순위)** 에 정렬됩니다.
     - 뱃지 디자인: `bg-blue-500` (파란색 등 눈에 띄는 상태 컬러), 텍스트 `IN PROGRESS`.

## 2. 데이터 모델 표준 (Data Model Standard)

상태 관리는 Firestore의 `likes` (위시리스트) 컬렉션과 `orders` 컬렉션을 연계하여 수행합니다.

### 2.1 ProductLike (위시리스트)
위시리스트는 단순히 좋아요 여부만 저장하는 것이 아니라, 해당 상품에 대한 사용자의 **현재 파이프라인 상태**를 관리하는 허브 역할을 합니다.

```typescript
export interface ProductLike {
  id: string;             // 문서 ID: {userId}_{productId}
  userId: string;
  productId: string;
  status?: 'liked' | 'pending' | 'in_progress'; // 비즈니스 파이프라인 상태
  createdAt: Timestamp;
  updatedAt?: Timestamp;
}
```

### 2.2 비즈니스 로직 스탠다드 (Service Layer)
각 상태로 전이하기 위한 서비스(Firebase Service) 메서드는 분리하여 명확한 책임을 갖도록 구현합니다.

*   `toggleLike()` : `liked` 상태를 토글하거나 문서를 생성.
*   `setProductPendingStatus()` : 상태를 `pending`으로 명시적 업데이트 (채팅 시작 시 호출).
*   `setProductInProgressStatus()` : 상태를 `in_progress`로 명시적 업데이트 (주문 완료 시 호출).

*(참고: 서비스 레이어 코드는 데이터 무결성을 위해 `setDoc` 시 존재 여부를 확인하고, 존재할 경우 `updateDoc`으로 `status`만 변경하는 방식을 사용합니다.)*

## 3. UI/UX 및 인터랙션 가이드라인

### 3.1 상품 상세 페이지 (Product Detail)
*   **플로팅 액션 요소 위치:** 'Like'와 'Chat(댓글/대화)' 버튼은 이미지 슬라이더 **우측 하단 플로팅(`absolute bottom-6 right-4 flex-col`)** 으로 배치하여 디자인의 일관성을 맞추고 이미지 감상을 방해하지 않아야 합니다.
*   **불필요한 카운트 제거:** 복잡도를 줄이기 위해 디자인에 없는 View 카운트 등은 생략하고, Like, Chat 등 핵심 전환(Conversion) 요소만 노출합니다.

### 3.2 구매 흐름 간소화 (Purchase Flow)
*   **주문 요약 (Order Summary):** 
    *   사용자가 최종 결정을 내리는 화면이므로 정보의 과부하를 막아야 합니다.
    *   Fulfillment 세부 사항, Price Breakdown, Production Info 등 불필요한 레이아웃을 제거하고, 핵심인 **선택 상품 정보, 총 결제 금액**만 명확하게 보여줍니다.
    *   **연락처(Contact Number) 입력 생략:** 플랫폼 유저들은 이미 전화번호 인증을 완료한 상태이므로, 구매 시 연락처를 다시 묻지 않고 시스템에 저장된 인증 데이터를 판매자에게 자동으로 전달(Binding)하여 클릭을 최소화합니다(1-Click 구매 지향).
*   **완료 화면 (Post-Purchase):**
    *   결제가 끝난 후에는 사용자가 다음 행동을 고민하지 않도록 단일 목적지인 **'Go to Shopping' 버튼**만 배치합니다.
    *   동시에 백그라운드에서는 주문 데이터(`Order`)가 생성되고, 위시리스트 상태가 `in_progress`로 승급되며, 판매자에게 자동 메시지가 발송되어야 합니다.

### 3.3 상태 기반 FAB (Floating Action Button / Wishlist Tray)
*   사용자가 자신의 진행 상태를 한눈에 볼 수 있도록 상시 떠 있는(Floating) UI를 제공합니다.
*   **정렬 원칙 (우선순위):**
    1.  `in_progress` (가장 중요, 진행 중인 거래)
    2.  `pending` (협의 중인 거래)
    3.  `liked` (단순 관심 상품)
*   **헤더 요약 문구:**
    트레이의 상태 요약 텍스트는 우선순위가 높은 상태가 있을 때만 해당 카운트를 동적으로 노출합니다.
    *(예: `1 IN PROGRESS, 1 PENDING, 3 LIKED`)*
*   **플러스(+) 등록 버튼 통합 (C2C 전용):**
    Resale 등 사용자가 직접 상품을 등록해야 하는 도메인에서는 별도의 '글쓰기' 플로팅 버튼을 만들지 않고, 이 상태 기반 FAB 내부에 **플러스(+) 버튼을 통합**합니다. 이를 통해 사용자 행동 액션(내 활동 내역 확인 + 새로운 상품 등록)을 하나의 허브(Action Hub)로 일원화하고 화면의 시각적 피로도를 낮춥니다.

## 4. 향후 타 서비스 적용 시 체크리스트

Stay(숙박), Rental(대여), Class(클래스) 등의 서비스를 신규 구축할 때 이 스탠다드를 동일하게 적용합니다.

- [ ] 해당 도메인의 위시리스트 데이터 구조에 `status` (`liked`, `pending`, `in_progress`) 파이프라인이 포함되어 있는가?
- [ ] 채팅/문의하기 버튼 클릭 시 상태가 `pending`으로 승급되는 로직이 연결되어 있는가?
- [ ] 예약/결제 완료 시 상태가 `in_progress`로 승급되는 로직이 연결되어 있는가?
- [ ] 글로벌 네비게이션 또는 FAB 메뉴에서 위의 세 가지 상태가 우선순위에 맞게 시각적으로 차등 정렬되어 노출되는가?
- [ ] 완료 페이지(Complete Step)에서 사용자 이탈을 막고 플랫폼 체류를 유도하는 단일 액션(예: '홈으로', '계속 탐색하기')만 노출되고 있는가?

---

이 스탠다드는 플랫폼 내 모든 서비스의 거래 경험을 통일하고, 직관적인 UX를 제공하기 위한 핵심 지침입니다. 코드를 리팩토링하거나 새로운 모듈을 설계할 때 항상 우선적으로 고려되어야 합니다.
